# 远程开发指南

> **完整工作流、详细规则、命令示例与 PowerShell 转义备忘见 `remote-dev` SKILL。** 本文件仅保留核心原则与工作流主干。

## 何时使用

- 任务涉及远端硬件（相机、传感器、机械臂等）、ROS2 联调、BPU/GPU/ARM 专有环境
- 用户明确要求"上板测试""推到板端""远端部署""SSH 登录"

## 核心原则

1. **上下文优先** — 连接远端前，先从会话、项目文档中收集主机/用户/目录/环境信息
2. **敏感信息不收集** — 密码、令牌、密钥必须让用户在终端直接输入
3. **SSH 免密复用** — 已有免密直接复用，不重复要求配置
4. **本地编辑，远端验证** — 代码本地写，仅必须依赖远端硬件的步骤放远端
5. **Git 优先 / 无 Git 先备份** — 有 Git 仓库走 Git 同步；无 Git 时修改前先完整备份

## 工作流主干（7 步，不可跳过）

| 步骤 | 要点 |
|------|------|
| 1. 上下文收集 | 从会话、README、DEVELOPMENT、SSH config 补齐远端信息 |
| 2. 判断执行位置 | 静态代码/文档 → 本地；硬件/ROS2 联调 → 远端 |
| 3. SSH 接入验证 | 先 `whoami && hostname && uname -a` 验证连通 |
| 4. 远端环境初始化 | 确认工作目录，source ROS2/虚拟环境 |
| 5. 本地开发 + 远端验证 | 按「开发模式」（A: Git全流程 / B: Git+指令推送）同步 |
| 6. 测试与调试 | 同步后先最小验证，优先本机可完成测试 |
| 7. 结果处理 | 数据采集远端，分析处理本地 |

## 开发模式

执行前必须从 `INFORMATION.instructions.md`或`基本信息.instructions.md` 读取「开发模式」字段：

- **模式A**：Git 全流程 — `git push/pull` 同步，严禁绕过 Git 直接复制文件
  - **测试分支规则**：AI 自主进行的任何提交和推送都必须在临时分支（如 `test/xxx-feature`）上进行，不得直接修改主分支或正式分支。测试顺利完成后，由用户手动合并回正式分支。AI 在会话结束时提供 squash 合并命令（如 `git merge --squash test/xxx-feature`）供用户参考执行。
- **模式B**：Git 版本管理 + 指令推送 — Git 做版本备份，`scp`/`rsync` 等指令推送到远端

## 关键行为红线

- **远端命令超时**必须主动 `kill_terminal`，不得等用户介入
- **AI 自主发起的 SSH 命令**必须设 `timeout`（推荐 15s）；用户交互类命令设 `timeout=0`
- **修改远端文件前**先备份并说明回退方式
- **每一步标注执行位置**（本地 / 远端）
- **排障顺序**：网络 → SSH → 身份 → 目录 → 环境 → 设备链路
