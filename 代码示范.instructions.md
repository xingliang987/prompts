# 代码风格底线规则

> 详细规范、完整模板、反例对照见 `coding-method` SKILL。本文件仅保留不可违反的底线。

## 通用

- 注释用**简体中文**，命名用**英文**（语义清晰，避免缩写）
- 注释说明"**为什么**"而非"是什么"；禁止零散短语拼贴；禁止模糊指代（"这里""那个"）
- 每个文件只做一件事，超过 300 行考虑拆分
- 错误路径必须显式处理，不允许空 catch / 裸 return

## 注释底线

- **所有类、所有函数/方法（含私有）必须有注释或 docstring**
- 文件头必须含"文件名 + 功能说明"
- 模块分区用 `# ---- 区块名 ----`（Python）或 `// ---- 区块名 ----`（C++），同文件不得混用

## 结构底线

- `__init__` 超过 15 行必须拆分为 `_init_xxx()` 私有方法
- 状态读写必须通过专用方法（`_set_state` / `_get_state`），禁止直接赋值 `self._state`
- 循环体内核逻辑超过 10 行必须提取为独立私有方法

## 命名底线

- 类 `PascalCase`，函数/变量 `snake_case`，常量 `ALL_CAPS`
- Python 私有方法加 `_` 前缀，初始化子方法以 `_init_` 开头
- C++ 实现代码包裹在业务命名空间内

## 日志底线

- 消息头必须有 `[分类标签]`（如 `[推理失败]` 而非 `[错误]`）
- 外部可降级失败用 `warning`，需人工介入才用 `error`
- 多阶段流程至少覆盖"开始 → 关键分支 → 下游调用 → 返回结果/失败原因"
- 失败日志必须带步骤名、依赖名、错误码/异常类型
- 日志正文优先键值上下文（`step=xxx error_code=-1`），禁止纯自然语言长句
